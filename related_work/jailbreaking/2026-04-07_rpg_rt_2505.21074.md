- Date checked: 2026-04-07
- Primary sources:

## 1. 한 줄 요약
이 논문은 **상용 black-box T2I 시스템의 “정체불명의 복합 방어기제”를 직접 가정한 뒤**, LLM이 프롬프트를 반복 수정하고, 시스템 피드백을 **선호(preference)** 로 모델링하여 다시 LLM을 DPO로 미세조정하는 **iterative prompt attack / red-teaming 루프**를 제안한다. 세 논문 중에서, 사용자가 말한 “evaluator + feedback 기반으로 prompt를 깎아가는 구조”에 **가장 정확하게 들어맞는 논문**이다.

## 2. 문제 설정
저자들이 겨냥하는 핵심 문제는 다음이다.

1. 기존 white-box red-teaming은 내부 모델 접근이 필요하다.
2. 기존 black-box 공격도 실제로는 특정 defense mechanism(예: detector 기반인지, removal 기반인지)을 어느 정도 알고 있다고 가정하는 경우가 많다.
3. 그런데 실제 상용 API는 defense가 **복합적이고 비공개**다.
4. 따라서 “어떤 방어가 들어있는지 모르는 상태”에서 **반복 query를 통해 방어기제를 학습**하는 red-teaming이 필요하다.

즉, 이 논문의 핵심 포지션은 단순히 “unsafe prompt를 잘 만들자”가 아니라,
**"반복적인 상호작용을 통해 target system의 잠재적 선호/거부 패턴을 학습하는 adaptive attacker"** 에 가깝다.

## 3. 제안 방법: RPG-RT
논문은 방법을 **RPG-RT (Rule-based Preference modeling Guided Red-Teaming)** 라고 부른다. 구조는 크게 3단계다.

### Stage 1. LLM prompt modification + query
- 원본 프롬프트를 LLM이 여러 개 변형한다.
- 각 변형 프롬프트를 target T2I black-box system에 질의한다.
- 시스템 반응은 대략 세 종류로 나뉜다.
  - **TYPE-1**: reject
  - **TYPE-2**: 통과는 했지만 detector 기준 SFW
  - **TYPE-3**: 통과했고 detector 기준 NSFW

### Stage 2. Rule-based preference modeling
논문이 흥미로운 부분은 여기다. 단순히 “NSFW면 좋다”가 아니라, query 결과를 선호관계로 바꾼다.

- 기본적으로 TYPE-3가 가장 선호됨
- TYPE-1, TYPE-2는 TYPE-3보다 덜 선호됨
- 하지만 TYPE-1(reject)을 TYPE-2보다 무조건 나쁘다고 보지는 않음
  - reject는 오히려 target이 unsafe semantics를 감지했다는 신호일 수 있기 때문
- 그리고 TYPE-2/TYPE-3 내부에서는 단순 binary가 아니라, **rule-based score**로 finer-grained ranking을 만든다.

### Stage 3. DPO fine-tuning
- 위에서 얻은 preference pair를 사용해 LLM을 **DPO(Direct Preference Optimization)** 로 업데이트한다.
- 즉, target system의 방어 패턴을 간접적으로 학습하도록 LLM을 tuning한다.
- 이후 이 fine-tuned LLM이 다음 iteration에서 더 잘 우회하는 prompt를 만든다.

## 4. 왜 단순 evaluator loop와 다른가?
표면적으로는 “evaluator-based prompt optimization”처럼 보이지만, 실제로는 몇 가지가 더 있다.

### 4.1 단순 점수 최대화가 아니라 preference learning
일반적인 evaluator loop는 보통
- prompt 생성
- score 부여
- score 높은 prompt 유지
정도에서 끝난다.

RPG-RT는 여기서 한 단계 더 나아가,
- coarse detector feedback
- harmfulness scoring
- innocuous semantics 보정
을 합쳐 **pairwise preference dataset** 을 만든 뒤,
LLM 자체를 target system에 적응시키는 방식으로 간다.

즉, 이 논문은 단순 검색(search)보다는 **online adaptation** 성격이 강하다.

### 4.2 semantic fidelity를 분리해서 다룸
저자들은 CLIP similarity만 쓰면 harmful semantics와 harmless semantics가 섞여 혼동된다고 본다.
그래서 representation을
- `f_n`: harmful content 축
- `f_s`: harmless semantics 축
으로 분리하는 scoring model을 학습한다.

이 scoring model은 대략 다음을 하도록 설계된다.
- NSFW intensity ranking
- benign semantics invariance
- benign semantics similarity alignment
- 원래 CLIP 정보 reconstructability 유지

이건 “그냥 공격 성공률만 높이는 prompt search”보다,
**유해성 증가와 의미 보존을 동시에 다루는 구조적 evaluator 설계**라고 볼 수 있다.

## 5. 실험 설정에서 중요한 점
논문은 19개 T2I 시스템, 여러 defense 타입, 3개 상용 API, 그리고 T2V까지 평가한다.

특히 defense 범주를 다음처럼 폭넓게 잡는다.
- detection-based defenses
- removal-based defenses
- safety alignment
- multiple defenses
- online commercial APIs

이 설정은 논문 메시지와 일치한다.
즉, **특정 방어를 아는 공격**이 아니라,
**여러 방어가 섞인 현실적 black-box 환경**을 노린다.

## 6. 핵심 결과
### 6.1 온라인 API 결과가 특히 중요
논문 본문 Table 4 기준:
- **DALL-E 3**: RPG-RT ASR **31.33%**
  - Sneaky 4.67, Ring 0.67, FLIRT 0.00
- **Leonardo.ai**: RPG-RT ASR **67.67%**
  - Sneaky 22.67, Ring 7.33, FLIRT 13.33
- **SDXL API**: RPG-RT ASR **20.33%**
  - Sneaky 11.67, Ring 6.00, FLIRT 0.00

저자 주장대로, 적어도 온라인 API 결과만 보면 **RPG-RT가 타 baseline보다 훨씬 강하다**.
특히 DALL-E 3에서 차이가 매우 크다.

### 6.2 논문이 내세우는 메시지
- 거의 모든 T2I 시스템에서 높은 ASR
- semantic similarity도 경쟁력 유지
- unseen prompt에도 transferability를 보임
- T2V로도 확장 가능

즉, 이 논문은 단지 “한 모델 뚫렸다”가 아니라,
**feedback-driven adaptive red-teaming이 실제로 범용 black-box 공격 프레임워크가 될 수 있다**는 걸 밀고 있다.

## 7. 이 논문의 강점
### 7.1 사용자의 직관과 가장 잘 맞음
사용자 코멘트처럼, 요즘 jailbreak 구조에서 흔한 건
- evaluator 두기
- feedback 모으기
- prompt를 iterative하게 refine하기
인데,
이 논문이 바로 그 흐름에 가장 가깝다.

### 7.2 “unknown defenses”를 직접 문제로 삼음
많은 논문은 사실상 특정 방어를 겨냥한다.
반면 RPG-RT는 애초에 **방어를 모르는 상황** 자체를 모델링한다.
이건 상용 API red-teaming이라는 현실 설정에서 매우 중요하다.

### 7.3 LLM을 static generator가 아니라 adaptive attacker로 사용
LLM을 단순 paraphraser가 아니라,
**query-feedback preference loop를 통해 target-specific attacker로 진화**시키는 구조가 설득력 있다.

---

## 8. 약점 / 비판 포인트
### 8.1 query budget / 실전 비용 문제
iterative query + DPO fine-tuning 구조는 강력하지만,
실제 상용 API에서
- 비용
- rate limit
- 계정 정지 위험
- 시간
이 클 수 있다.

즉 성능은 좋지만, **실전 운영비용이 낮은 공격**인지는 별도 문제다.

### 8.2 detector / scoring 설계 의존성
논문은 coarse feedback를 보완하려고 scoring model을 도입했지만,
그 scoring 설계 자체가 공격 품질에 상당히 영향을 줄 가능성이 있다.
즉 “unknown defense를 잘 다룬다”는 장점과 별개로,
**공격자가 가진 외부 evaluator 품질에 성능이 좌우될 수 있음**.

### 8.3 prompt-space 중심 한계
아무리 adaptive해도 결국 주요 조작 대상은 **prompt** 다.
따라서 내부 latent/state를 직접 건드리는 공격(예: IVO)과 비교하면,
내부 unlearning defense가 강한 모델에서는 ceiling이 있을 수 있다.

## 9. 우리 연구 맥락에서의 해석
이 논문은 다음 질문에 가장 직접적인 답을 준다.

> “diffusion jailbreak에서 guidance 자체를 건드리기보다, prompt + evaluator + feedback loop가 더 메인스트림인가?”

내 해석은 **그렇다** 쪽에 가깝다.
특히 black-box / commercial API 관점에서는,
- 내부 guidance 접근 불가
- 내부 state 접근 불가
- defense 구조 미공개
이므로,
가장 현실적인 공격 surface는 결국 **prompt refinement loop** 다.

RPG-RT는 그 흐름의 정교한 버전이다.

## 10. 다른 두 논문과의 관계
### vs JPA (2404.02928)
- JPA는 **CLIP embedding space에서 NSFW concept direction을 찾고**, prefix prompt를 discrete optimization으로 만든다.
- RPG-RT는 **iterative query feedback + preference modeling + DPO** 다.
- 따라서 JPA가 더 “embedding-guided prompt optimization”이라면,
  RPG-RT는 더 “feedback-adaptive agentic prompt optimization”이다.

### vs IVO (2602.00175)
- IVO는 prompt보다 **initial latent optimization** 에 무게 중심이 있다.
- 특히 target이 **unlearned diffusion model** 일 때, 잔존 latent memory를 깨우는 방향이다.
- 따라서 RPG-RT는 일반 black-box red-teaming,
  IVO는 unlearning robustness를 깨는 latent-space attack이라는 차이가 크다.

---

## 11. 최종 메모
세 논문 중에서,
- **현재 mainstream jailbreak 감각**에 가장 가까운 것은 **RPG-RT**
- 이유는 prompt를 직접 계속 수정하고,
- evaluator/feedback를 구조화하고,
- 그 feedback으로 attacker 자체를 업데이트하기 때문.

즉, 사용자의 직관인
**“대부분 evaluator를 두고 feedback 기반으로 attack prompt를 계속 깎는 구조 같다”**
라는 관찰을 가장 강하게 지지하는 논문이다.
